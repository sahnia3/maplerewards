1 · Executive summary

This assessment combines a live mobile-browser QA sweep of all major routes, a 35-route static responsive audit, a Chrome-extension stress test against the live API, and an infrastructure/deployment review. The findings below separate Fixed (done & verified this session) from Blocker / High items that remain.

The four launch blockers (must fix before charging anyone)

  1. Stripe price IDs are documented under the wrong names → Pro checkout fails for everyone. Checkout reads STRIPE_PRICE_ID_PRO_ANNUAL / PROPLUS_ANNUAL / LIFETIME, but SHIP.md and .env.example only tell you to set MONTHLY / ANNUAL / LIFETIME. Follow the runbook and the default "Pro" checkout throws "stripe price ID not configured". internal/service/billing.go:84,109,131 · SHIP.md:38
  2. Cross-domain auth cookies are dropped by Safari / iOS. The app (vercel.app) and API (railway.app) are different registrable domains, so the API's auth/session/CSRF cookies are third-party (SameSite=None) and silently blocked by Safari ITP + any "block third-party cookies" setting — a large share of Canadian (iPhone-heavy) users can't stay logged in. The codebase already supports the fix via COOKIE_DOMAIN; it needs a shared parent domain (e.g. app.maplerewards.ca + api.maplerewards.ca), which is a DNS + env change, not a rewrite. internal/handler/auth_cookies.go:37-47
  3. The background worker may not be running. cmd/worker is built into the image but must run as its own Railway service. If it isn't: the PIPEDA 30-day account-deletion purge never executes (a compliance promise), and award-watch alerts, weekly digests, offer-expiry, promo-sentinel and CPP refresh — advertised Pro value — silently never fire. cmd/worker/main.go · Dockerfile
  4. The Chrome extension does not work in production. It points at localhost + a maplerewards.app domain that isn't where prod lives, its marquee "best card on checkout" bar never renders for a normal user, and once a user signs in the extension can't read their (owner-scoped) wallet at all. See §4. browser-extension/*

3 · Mobile parity

Status: The worst breaks are fixed and verified; a set of data-dense pages still clip columns on narrow phones. The app never shows a horizontal scrollbar (the scroll root uses overflow-x:clip), so the failure mode is "right-hand columns cut off," not "ugly scroll."

Fixed & verified Done

  • Chat input usable on mobile (was hidden behind nav) · /compare stacks · card-fan fits · iOS-zoom inputs · landing grid · loyalty tiles.
  • Swept clean on mobile (no overflow): /optimizer, /wallet, /cards, /pro-tools, /insights (index), /pricing, /settings, /trip-planner.

Remaining mobile punch-list (high) — data-dense fixed-px grids that clip on ≤430px

Each is a fixed multi-column grid with no mobile breakpoint; the recommended fix is to collapse to a single column or wrap in a horizontal-scroll container below 560–768px.

components/editorial/card-fan.tsx:291-301,371 (used on home)
All elevation/selection is onMouseMove/Enter; nothing fires on touch, so phone users get no feedback and tapping just navigates to /cards.
Fix: add onTouchStart/End handlers + a "tap a card" affordance, or auto-rotate focus on small screens.
High Ledger/row grids overflow on phones
cards/page.tsx:450 · insights/page.tsx:261,392,446 · portfolio/page.tsx:221,241 · components/optimizer-form.tsx:703 · onboarding/page.tsx:295

4 · Chrome extension — stress-test results

What it's supposed to do (per manifest + store listing): on 25 Canadian retailer checkouts, show a floating "best card for this purchase" bar; in the toolbar popup, show your wallet; warn when a store doesn't take Amex.

Verdict: a well-architected MVP scaffold that does not work end-to-end in production. The API contract and the security of its fetch-proxy are correct; the auth/session model and prod host config are not wired for real users.

What actually works Verified

  • API parsing is correct. Against the live API, an anonymous wallet's GET /wallet/{id} and POST /optimize return exactly the shapes the popup and content script parse (card.name/point_balance; card_name/program_name/effective_return). The Amex-blackout warning logic is sound.
  • The background fetch-proxy is production-grade security. Strict sender.id check, empty externally_connectable, and an anchored method+path allow-list (only GET ^/wallet/[a-f0-9]{32}$ and POST ^/optimize$) — no IDOR/SSRF surface beyond the two intended calls.
  • Icons are real, correctly-sized branded PNGs (16/32/48/128) — submission-ready (the README/STORE-LISTING calling them "placeholders" is stale).

What's broken / blocks production

Blocker Can't reach production at all
manifest.json:8-12 · background.js:11
DEFAULT_API_BASE is http://localhost:8080 and host_permissions lists only localhost + maplerewards.app / *.maplerewards.app — none of which is the real prod host (maplerewards-production.up.railway.app / maplerewards.vercel.app). A published extension can't call the API.
Fix: add the real prod hosts to host_permissions and set apiBase to the prod API (ideally the future api.maplerewards.ca).
Blocker The marquee floating bar never renders for a normal user
content.js:47-55,77-83 · popup.js:46-75
The content script reads the session only from chrome.storage.local.mr_session_id — which is never populated automatically. The popup's cookie-bridge reads a mr_session cookie that the backend never sets (verified), then falls back to a value you must paste in by hand via DevTools. So the headline feature requires manual setup that no real user will do.
Fix: build the real session bridge — have the web app postMessage the session id to the extension on login (or set a readable cookie on the shared parent domain), and make content.js use it.
Blocker The access model contradicts the listing the moment you sign in
content.js / popup.js (cookie-only auth) vs RequireSessionOwner
The extension authenticates only via cookies. Once a user signs up, their wallet is owner-scoped and the API returns 404 to any non-authenticated caller — and the web app's access token lives in memory, not a cookie the extension can send. So the extension only ever works for purely anonymous wallets, while the app funnels everyone into sign-up.
Fix: bridge a real auth token to the extension, or add an extension-specific scoped token endpoint.

Recommendation: treat the extension as pre-launch. It is not ready to publish to the Chrome Web Store; ship the web app first, then do the session-bridge + prod-host work as a dedicated follow-up. Do not list it as a live feature when charging customers.

5 · Infrastructure & deployment

Blockers / high

Blocker Stripe tier price IDs unset if you follow the runbook → nobody can buy Pro
internal/service/billing.go:84-86,109,131 · SHIP.md:38 · .env.example:51-52
Current checkout sends pro_annual/proplus_annual/lifetime → reads STRIPE_PRICE_ID_PRO_ANNUAL / PROPLUS_ANNUAL / LIFETIME. The docs only mention MONTHLY/ANNUAL/LIFETIME, so those tier vars stay empty and checkout throws "price ID not configured."
Fix (your action): on Railway set STRIPE_PRICE_ID_PRO_ANNUAL, STRIPE_PRICE_ID_PROPLUS_ANNUAL, STRIPE_PRICE_ID_LIFETIME to the real price IDs (test IDs in sandbox, live IDs at cutover). Also subscribe the webhook to charge.refunded so refunders lose Pro. Update SHIP.md/.env.example.
Blocker Cross-domain third-party auth cookies (Safari/iOS drop them)
internal/handler/auth_cookies.go:37-47 · csrf.go
vercel.app ↔ railway.app are different sites, so session/refresh/CSRF cookies are third-party SameSite=None and silently blocked by Safari ITP + third-party-cookie blocking. The body-token CSRF fallback still needs the cookie to round-trip.
Fix (your action): put app + API on one parent domain — e.g. app.maplerewards.ca (Vercel) + api.maplerewards.ca (Railway) — and set COOKIE_DOMAIN=.maplerewards.ca. The code already emits SameSite=Lax; Domain=… when COOKIE_DOMAIN is set, so this is DNS + one env var. (This also fixes the extension cookie-bridge.)
Blocker Background worker not confirmed running
cmd/worker/main.go · Dockerfile
If ./worker isn't a running Railway service: PIPEDA 30-day deletion purge never runs (compliance), and award-watch alerts / digests / offer-expiry / promo-sentinel / CPP refresh never fire.
Fix (your action): add a second Railway service from the same image with start command ./worker and the same DB/Redis/API-key env; confirm its logs show the sweep loop.
High Redis env footgun + manual migrations
cmd/api/main.go (REDIS_ADDR/PASSWORD) · SHIP.md
Code reads REDIS_ADDR/REDIS_PASSWORD but Railway hands out REDIS_URL — a mismatch silently falls back to localhost and fails to boot. Migrations are fully manual with no startup guard, risking serving against an old schema.
Fix: map REDIS_ADDR from Railway's value (or parse REDIS_URL); run migrate up as a release step and/or add a startup schema-version check.

What's genuinely solid Do not touch

  • Stripe webhook: fail-closed on missing secret, bounded body, constant-time HMAC-SHA256, two-sided 5-min timestamp tolerance, and a RESERVED→COMPLETED idempotency state machine with stale-orphan reclaim (409 keeps Stripe retrying). handler/billing.go
  • CORS: exactly one configured origin echoed (never *), credentials + fixed allow-list, and prod boot refuses empty/*/non-https origins.
  • Boot gates: JWT_SECRET (≥32 chars, non-default), CORS_ORIGIN, STRIPE_WEBHOOK_SECRET, REDIS_PASSWORD all os.Exit(1) in production if missing/invalid.
  • Data: all SQL parameterized; JWT rejects non-HMAC alg; CSRF double-submit; IDOR guards on session-scoped routes.
